# BA2_check
# Проект: агент-перепроверщик .cha расшифровок (курс «Продвинутый ИИ», Трек 2)

## Задача курса
Трек 2 «ИИ-агент»: воркфлоу с языковой моделью (минимум 3 шага), собранный в n8n / nodul / Яндекс Клауде.
Бонус: память агента, RAG, внешние инструменты.

## Наша задача
В корпусе детской речи (мастерская CHILDES) студенты вручную перепроверяют
расшифровки .cha, созданные автоматически (Batchalign).
Даже после ручной проверки остаются систематические ошибки — это видно по файлам
«Типичные ошибки.docx» каждого проверяющего (см. `docs/typical_errors.txt`).

Агент должен: взять уже вручную проверенный .cha файл → перепроверить морфологическую (%mor)
и синтаксическую (%gra) разметку по правилам курса → выдать:
1. чистый исправленный .cha файл;
2. отдельный файл с комментариями — что и где исправлено

## Формат .cha (кратко)
- `%mor:` — морфология: `pos|лемма-Категория1-Категория2-...` для каждого слова по порядку.
- `%gra:` — синтаксис: `номер|номер_вершины|ТЭГ` (тэги в файлах реально в UPPERCASE: ROOT, NSUBJ, OBJ, PUNCT...).
- Ровно один `ROOT` на предложение, его вершина — `0`.
- Порядок морф. категорий: сущ/прил — род-число-падеж; глагол — финитность-вид-наклонение-время-число(+лицо).

Правила и частые ошибки извлечены в:
- `docs/extracted_docs.txt` — инструкции + FAQ разметчиков
- `docs/typical_errors.txt` — реальный каталог ошибок по всем проверяющим (используем как тест-кейсы)

## Архитектура

Прототип и тесты правил — локально (Python, `prototype/`), финальный воркфлоу — n8n (`n8n/workflow.json`).

Шаги воркфлоу:
1. **Вход**: чтение .cha файла (Read Binary File / из папки)
2. **Code (JS) — детерминированные правила**: парсинг .cha → разбор по предложениям,
   проверки без LLM (см. `prototype/rules.py` — обнаруженные категории ошибок):
   - пропущенные категории (род/число/падеж у сущ/прил, вид/время/наклонение у глаг.)
   - число у инфинитива (не должно быть)
   - ровно один ROOT с вершиной 0 на предложение
   - табы vs пробелы после `%mor:`/`%gra:`
   - согласование числительных 2-4 (Sing-Gen, не Plur-Nom)
   - эвристика Nom/Acc при роли OBJ
   - пропущенный Voc при обращении (VOCATIVE)
   на выходе — список подозрений по каждому предложению
3. **RAG (мини-база знаний)**: инструкции + FAQ + типичные ошибки как источник контекста
   для LLM-узла (bonus-критерий курса)
4. **LLM-узел (агент)**: для подозрительных предложений — подтвердить/отклонить, предложить
   исправленную строку %mor/%gra, дать короткий комментарий с обоснованием
5. **Code — сборка**: собрать исправленный .cha (не трогая остальные тиры) + файл комментариев
6. **Выход**: два файла — `{имя}-rechecked.cha` и `{имя}-comments.md`

до включения одушевлённости) — 0 ошибок парсинга на 576 файлах / 156k предложений.

**Одушевлённость (Anim/Inan)** — отдельная история: эмпирически проверили, что даже в уже
проверенных вручную файлах эта категория проставлена только у ~2.5% существительных
(825 из 32710). Это не единичный пропуск, а системная недомаркировка по всему корпусу.
Решение: включить как правило **обогащения** разметки (не "ошибка", а задача "определи
и добавь"), сознательно приняв рост объёма обработки (~x3 LLM-вызовов на файл) — решение
принято автором проекта явно, с пониманием этого компромисса. Заодно эта категория используется
для уточнения `case_role_mismatch`: одушевлённые (Anim) существительные мужского рода
и множественного числа тоже надёжно различают Nom/Acc (Acc=Gen), в отличие от
неодушевлённых, где Acc=Nom по форме.

## Ещё две доработки: порядок категорий, опечатки и лемма

**`noun_category_order`** — порядок категорий у сущ. должен быть род-число-падеж
(-одушевлённость). Проверено на всём корпусе: 63827 из ~70000 существительных строго
следуют этому порядку — отклонения либо уже ловятся `noun_missing_category` (пропуск),
либо это реальная ошибка порядка. Найдено **50 случаев** на весь корпус (например
"узор": C-N-C вместо N-C) — точное, малошумное правило.

**Опечатки расшифровщика и корректность леммы** — в отличие от остальных правил, это не
отдельный триггер, а расширение того, что LLM-узел проверяет в уже присланном ему
предложении: (1) явные опечатки НАБОРА ТЕКСТА (не путать с настоящим произношением
ребёнка — те сохраняются как есть), (2) лемма должна быть начальной формой слова
(инфинитив у глаголов, а не спрягаемая форма). Технический нюанс: реплику нельзя
перепечатывать целиком — в файлах с `@Options: bullets` есть невидимые служебные символы
привязки к видео/аудио, которые модель при перепечатке может потерять. Поэтому опечатки
в реплике модель возвращает как точечные замены `{from, to}`, а не новую строку целиком —
код сам подставляет замену в исходный текст, не трогая остальное.

## Проверка согласованности (без LLM) + честный предел автоматизации

LLM обрабатывает каждое предложение отдельным независимым запросом и не помнит, что
решила для того же слова в другом предложении того же файла — из-за этого повторяющееся
слово (например имя) иногда получает категорию (например одушевлённость), а иногда нет.
Добавлен детерминированный проход **после** LLM (в `rebuild_and_report.js`): собирает уже
определённые соответствия лемма+часть речи → Anim/Inan по всему файлу и подставляет их
туда, где пропущено — без LLM, гарантированно. Проверено на реальном примере ("дракон"),
работает корректно.

**Важно понимать предел этого механизма**: он лечит именно НЕСОГЛАСОВАННОСТЬ повторов —
если слово встречается в файле несколько раз и хотя бы один раз размечено правильно,
остальные подтянутся. Но если слово встречается **один раз** и модель его пропустила —
подстраховки нет, кроме самой модели.

### Для единичных пропусков: верификация + один точечный повтор

Добавлена вторая линия защиты для случаев, когда слово встречается один раз (проход
согласованности здесь не поможет). В `llm_recheck.js`:
1. После ответа модели код **сам** (без LLM, чисто на правилах) перепроверяет каждое
   исходно найденное подозрение по факту — правда ли оно решено в исправленном %mor/%gra.
   Раньше мы просто доверяли полю `hasRealIssue`, теперь — проверяем результат напрямую.
2. Если что-то осталось нерешённым — делается **один дополнительный точечный запрос**
   именно по этому остатку (не по всему предложению заново), с текущим (уже частично
   исправленным) состоянием как отправной точкой.
3. Если и это не помогло — предложение попадает в новый раздел отчёта
   **"Требует ручной проверки"** в `{файл}-comments.md`, с указанием, какое именно
   правило не подтвердилось. Это не решает проблему единичных пропусков полностью
   (LLM может не справиться и со второй попытки), но: (а) заметно сокращает их долю за
   счёт повторной попытки, и (б) — что важнее — переводит человеческую проверку из
   "перечитать весь файл" в "проверить именно эти N конкретных строк из отчёта".

При тестировании нашёлся и настоящий баг в самой проверке (не в LLM): отчёт сверял
"решено или нет" по номеру строки из **исходного** файла, а после того как более ранние
многострочные теги схлопывались в одну строку, все номера ниже съезжали — часть уже
решённых пунктов ошибочно попадала в "требует проверки". Починено (пересчёт номеров строк
с учётом уже применённых правок, `translateIndex` в `rebuild_and_report.js`).

### Проверили ещё одну идею для token_count_mismatch — и отклонили её

Обе оставшиеся после второй попытки LLM ошибки `token_count_mismatch` в T27.cha — это
случаи, где в конце предложения не хватает ровно одной записи в `%gra` для финального
знака препинания. По всему корпусу такой знак чаще всего крепится к ROOT (72.9% случаев).
Проверили уточнённую эвристику — крепить к последнему из цепочки ROOT+однородные (CONJ) —
точность выросла лишь до 77.8%. Это означает, что почти **каждое пятое** предложение
получило бы неверную синтаксическую связь при слепом автоматическом применении — для
данных, идущих в открытую базу, недопустимый риск. Решили НЕ делать это отдельным жёстким
правилом, а только добавить как мягкую подсказку в промпт LLM (со статистикой, но без
предписания) — итоговое решение остаётся на модели/человеке, а не на позиционной эвристике.

Верификация покрывает 15 из 16 правил-ошибок (все, кроме `case_role_mismatch`,
`numeral_2_4_agreement` и опечаток в тегах — там либо низкий объём, либо контекст
меняется настолько, что механическая перепроверка сама по себе ненадёжна).

Отдельно рассматривали правило для местоимений, ошибочно размеченных как другая часть
речи (например "тебя" как noun) — протестировали на корпусе и **не стали включать**:
из 329 похожих случаев подавляющее большинство — не ошибки, а легитимные варианты
("intj|я-я-я" — ребёнок повторяет звук как восклицание; "det|её-..." — притяжательное
"её" законно размечается как det, а не pron; "let|я" — буква алфавита). Включение дало
бы больше шума, чем пользы. Это подтверждает общий принцип: не все ошибки механически
отличимы от нормы без понимания контекста.

**Рекомендация перед публикацией в открытую базу**: ни это, ни любое другое
LLM-based решение не может гарантировать 0% ошибок — модель делает вероятностные
решения, и даже с проверками согласованности и всеми правилами часть решений может быть
неверной (как показали примеры выше). Для данных, которые пойдут в открытую
исследовательскую базу, финальный человеческий выборочный контроль перед публикацией
(хотя бы по нескольким файлам на партию) — не формальность, а реально нужный шаг,
особенно учитывая, что сам проект существует именно потому, что ручная проверка сама по
себе тоже не была безошибочной.

**Важная находка при тестировании**: правило `case_role_mismatch` изначально давало много
ложных срабатываний (1520 → 495 после уточнения) из-за падежного синкретизма русского
языка — у местоимений "что"/"это"/"всё" и у сущ. среднего/неодушевлённого мужского рода
формы Nom и Acc совпадают по написанию, это норма, не ошибка. Правило сузили: применяется
только там, где падежи гарантированно различаются (личные местоимения, сущ. женского
рода). Это тот же принцип, что и в остальном проекте — доверять только тому, что
подтверждено на реальных данных, а не первой гипотезе.

Опечатки в тегах (`gra_tag_typo`, `mor_pos_typo`) ловятся через эмпирический белый список
реально встречающихся в корпусе тегов (см. `docs/tag_frequency.txt`, посчитан по всем
576 файлам) + похожесть на них (расстояние Левенштейна ≤ 2) — так редкие, но настоящие
теги UD (DEP, COMPOUND, GOESWITH) не путаются с опечатками вроде PARATXIS.

Важный технический момент: строки `www .` / `xxx .` (нерасшифрованная/неразборчивая речь по
конвенции CHAT) не имеют %mor/%gra — это нормально, не ошибка. Правила это учитывают.

Вывод для архитектуры: детерминированный слой имеет высокую отдачу (recall) и почти не
даёт ложных срабатываний на структурных проверках. Он служит **фильтром кандидатов**:
не решает сам, а передаёт LLM-узлу короткий список подозрительных предложений с указанием,
что именно проверить — так экономим токены и не гоняем модель по всему файлу вслепую.

## Технические решения
- n8n: установлен локально (Node.js + `npx n8n`) — бесплатно навсегда, не нужен облачный
  тариф, воркфлоу запускается вручную и не требует круглосуточной работы сервера
- LLM: YandexGPT (Yandex Cloud Foundation Models, через Yandex AI Studio) — платформа явно
  разрешена заданием курса, доступна с российской картой/грантом, не требует иностранной
  оплаты (в отличие от OpenAI/Anthropic, которые изначально рассматривались, но отклонены
  по этой причине)
- Вывод: исправленный .cha + markdown-файл с комментариями по исправлениям (плюс бинарные
  вложения для скачивания прямо из вывода узла — `this.helpers.prepareBinaryData`)

## Roadmap: от перепроверки студентов к прямой проверке сырого BA2

Полная картина процесса в мастерской CHILDES — несколько этапов ручной проверки:
1. Batchalign (BA2) автоматически размечает сырую расшифровку
2. Часть I: студент вручную размечает/исправляет сырой BA2-вывод с нуля
3. Часть II: другой студент перепроверяет то, что сделал первый

**Текущая версия агента** работает на границе шага 3 — перепроверяет то, что уже
перепроверил человек (входные данные — «Проверенные файлы» из Часть II). Это то, что
реально протестировано и сдаётся по курсу.

**Следующий шаг (не в текущей сдаче)** — перевести агента на вход сразу после шага 1,
то есть перепроверять сырой вывод BA2 напрямую, до участия студента. Сравнение сырых
файлов Часть I (`Исходные файлы`) с исправленными студентом (`Исправленные файлы`)
показало, что профиль ошибок там принципиально другой и гораздо плотнее:

- **ROOT почти всегда замкнут сам на себя** (`3|3|ROOT` вместо `3|0|ROOT`) — это
  систематическая ошибка BA2, а не редкий случай, как в уже проверенных файлах
- **Междометия/звукоподражания массово размечаются как noun/verb/adj с абсурдными
  леммами**: `мм` → `noun|миллиметр`, `мммм` → `verb|мммать`, `Господи` →
  `verb|Господь-...-Imp`, `нет` → `verb|нет-...` вместо `part|нет`
- **Скобочные исправления `[: правильно]` иногда игнорируются** при разборе
- Почти каждое предложение требует вмешательства, а не единицы на файл

Это меняет архитектуру: текущий пайплайн «правила находят редких подозреваемых → LLM
точечно проверяет» рассчитан на *мало* ошибок. Для сырого BA2 нужен гибридный подход:

1. Часть находок (в первую очередь замкнутый на себя ROOT) можно чинить **механически,
   без LLM** — если в предложении один ROOT-тег и его вершина равна его же номеру,
   правило само переставляет вершину на 0
2. Новое правило `suspicious_interjection_pos` — эвристика по коротким
   словам-повторам (мм, ммx, xаxа...), размеченным не как intj — кандидат для LLM
3. Для оставшегося (лемма/POS для нечёткой речи, различение придаточных и т.п.) —
   LLM по-прежнему нужен, но уже на **каждое** предложение, а не на редкие подозрения —
   здесь и пригодится батчинг (несколько предложений на один запрос) для контроля
   стоимости и скорости

Осознанно отложено до следующей итерации — рабочий пайплайн для Часть II важнее
поставить стабильно к дедлайну курса, чем начинать более рискованную переделку.

**Важное ограничение для этого режима**: новые, ещё не анонимизированные расшифровки
детей — персональные данные несовершеннолетних (152-ФЗ). Обработка и разработка вокруг
таких данных должна оставаться в российской юрисдикции (локальный n8n + YandexGPT,
без Claude/OpenAI на реальных новых данных). Подробности и обоснование — см.
`docs/data_policy.md`.
