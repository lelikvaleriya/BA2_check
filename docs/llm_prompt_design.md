# Дизайн LLM-узла

## Зачем LLM, если есть правила
Правила (`n8n/rules_check.js`) детерминированно находят **кандидатов** на ошибку —
но не могут: (а) проверить лемму/POS на смысловую корректность, (б) решить, короткая
форма прилагательного или нет, (в) понять контекст многозначных слов, (г) написать
понятный комментарий на русском. Это и есть работа LLM — "осмысленное использование
языковой модели", которое требует критерий курса.

## Кто вызывает модель и как часто
LLM вызывается **по одному разу на подозрительное предложение** (не на весь файл целиком —
так дешевле и модель не отвлекается на остальные 90% нормального текста).
Batch по предложениям делается узлом Split In Batches в n8n.

## Системный промпт (для LLM/AI Agent узла)

```
Ты помогаешь перепроверять морфологическую (%mor) и синтаксическую (%gra) разметку
детской речи в формате CHILDES/CHAT, используя категории Universal Dependencies
(конвенции корпуса Taiga для русского языка).

У тебя есть инструмент поиска по базе знаний с правилами разметки — используй его,
если не уверен в правиле для конкретной категории (падеж, число, вид, ROOT и т.д.).

Тебе присылают: реплику, текущую разметку %mor и %gra, и список конкретных подозрений,
найденных автоматической проверкой правил. Твоя задача для каждого подозрения:
1. Проверить, является ли это реальной ошибкой (не все подозрения подтверждаются —
   иногда правило сработало ложно, например на нестандартном, но верном случае).
2. Если ошибка реальная — дать исправленную строку %mor и/или %gra целиком
   (со всеми токенами по порядку, не только исправленный).
3. Написать одно короткое предложение-комментарий на русском: что изменено и почему,
   в стиле существующих документов "Типичные ошибки" (например: "у глагола 'была'
   добавлен род Fem — прошедшее время требует согласования по роду").

Если предложение проверено и всё в порядке — верни его без изменений и не пиши комментарий.

Отвечай СТРОГО в формате JSON (см. схему ниже), без пояснений вне JSON.
```

## Формат входа (на одно подозрительное предложение)

```json
{
  "fileName": "T14-checked.cha",
  "lineNumber": 12,
  "textLine": "*MOT:\tну что , Тося , давай покажем , как мы на горку залезаем ! 1640_5570",
  "morRaw": "part|ну pron|что-IntRel-Nom-S1 cm|cm propn|Тося-Fem-Sing-Nom ...",
  "graRaw": "1|6|DISCOURSE 2|1|FIXED 3|2|PUNCT ...",
  "ruleFlaggedIssues": [
    { "rule": "interrogative_pronoun_has_number_person", "wordNo": 2,
      "detail": "У вопросительного/относительного местоимения 'что' указаны число/лицо" }
  ]
}
```

## Формат выхода (модель обязана вернуть)

```json
{
  "fileName": "T14-checked.cha",
  "lineNumber": 12,
  "hasRealIssue": true,
  "correctedMor": "part|ну pron|что-IntRel-Nom cm|cm propn|Тося-Fem-Sing-Nom ...",
  "correctedGra": "1|6|DISCOURSE 2|1|FIXED 3|2|PUNCT ...",
  "comment": "у местоимения 'что' убраны число и лицо (S1) — по правилам вопросительные/относительные местоимения не размечаются числом и лицом"
}
```

Если `hasRealIssue: false` — `correctedMor`/`correctedGra` совпадают с исходными,
`comment` — пустая строка (такие предложения не попадают в итоговый файл комментариев).

## RAG: как модель ищет контекст

**Реализовано (в `n8n/llm_recheck.js`)**: облегчённый вариант без embeddings — по
названию сработавшего правила (`rule`, его возвращают правила из `rules_check.js`)
код напрямую подставляет соответствующий кусок `n8n/knowledge_base.md` (захардкожен
как `RULE_CONTEXT` в коде) в промпт модели. Это по-прежнему retrieval + augmented
generation (находим релевantный документ → подмешиваем его в контекст генерации),
просто без векторного поиска — надёжнее для первого запуска и не требует отдельного
эмбеддинг-провайдера.

**Опциональный апгрейд (см. "Апгрейд: настоящий RAG" в `n8n/BUILD_GUIDE.md`)**:
`n8n/knowledge_base.md` разбит на чанки по темам (разделены `---`), можно загрузить
через **Embeddings → Default Data Loader (markdown, split by `---`) → Simple Vector
Store (in-memory, режим "Insert")** — один раз при первом запуске воркфлоу.
Дальше основной воркфлоу использует **Simple Vector Store (режим "Retrieve as Tool")**,
подключенный как инструмент к **AI Agent**-узлу — модель сама решает, когда стоит
"заглянуть" в конкретное правило. Выглядит "по классике" на скриншотах, но добавляет
риск на первом запуске — рекомендуется сначала сдать рабочую версию как есть.

## Сборка результата
Узел Code после LLM:
1. По каждому предложению с `hasRealIssue: true` — заменить `morRaw`/`graRaw` в исходном
   тексте файла на `correctedMor`/`correctedGra`, сохранив все остальные строки как есть.
2. Собрать все `comment` в отдельный markdown-файл `{имя}-comments.md`, сгруппировав
   по номеру строки — в стиле существующих "Типичные ошибки.docx".
3. Отдать на выход два файла: `{имя}-rechecked.cha` и `{имя}-comments.md`.
